# N42 90K Cap 压力测试 — 并发时序分析报告

**测试日期**: 2026-03-11
**环境**: Ubuntu, 7节点本地测试网
**配置**: N42_MAX_TXS_PER_BLOCK=90000, N42_FAST_PROPOSE=1, N42_DEFER_STATE_ROOT=1, N42_SKIP_TX_VERIFY=1, N42_INJECT_HIGH_WATER=90000

### 瓶颈排名:
1. **CompactBlock P2P传播: 1060ms (58%)** — 90K tx hash的序列化+网络传输+反序列化+从pool匹配
2. **Payload Build: 400ms (22%)** — EVM执行101ms + Pool开销70ms + 其他229ms
3. **R2 Vote收集: 157ms (9%)** — 等待 5/6 Commit投票
4. **R1 Vote收集: 72ms (4%)** — 等待 5/6 投票
5. **Commit写盘: ~50ms (3%)** — 区块持久化
6. **其他开销: ~81ms (4%)** — 消息序列化/反序列化等

## 九、结论

1. **90K txs/block 是稳定极限**: 链能持续稳定运行，无timeout，无崩溃
2. **瓶颈是CompactBlock P2P传播** (占58%)，不是EVM执行或打包
3. **Payload Cache完美工作**: 100%命中率，Follower完全跳过EVM重执行
4. **Pool Gate机制有效**: 维持pending=90K，无nonce gap，无交易驱逐
5. **FAST_PROPOSE模式下出块间隔~1.85s**: 因为CompactBlock传播需要~1.06s，加上Build的~400ms

### 优化方向:
- **压缩CompactBlock**: 当前传输90K tx_hash (90K × 32B = 2.8MB)，可考虑bloom filter或index编码
- **并行化Build**: EVM执行与Pool拉取可以pipeline化
- **增加Pacemaker Timeout**: 生产环境应设为 ≥4秒以容纳proposal延迟波动
